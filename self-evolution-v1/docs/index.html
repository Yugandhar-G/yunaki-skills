01the problem

Your agent makes Monday's mistake again on Tuesday.

Agents are stateless across runs. Every task starts from zero, and the only knobs are a bigger model or hand-tuned prompts that go stale. Fine-tuning is slow and expensive. Yesterday's lesson just evaporates.

without yunaki
  • Same failure repeats across runs
  • "Improvement" = a pricier model or manual prompt-tuning
  • Fine-tuning is slow, costly, opaque
  • What worked last time is gone next time
with yunaki
  • Each run's lesson becomes a versioned skill
  • The next run starts ahead — same model
  • No weights touched, no prompts babysat
  • Improvement compounds, measured against a control
02the loop

How it works

No fine-tuning. No prompt babysitting. Yunaki watches what the agent does, distills the lesson, and feeds it forward.

1

Run & observe

The agent attempts a task. It fails, partially succeeds, or passes — the full trace and eval score are captured.

trace + score → captured
2

Extract the skill

Yunaki distills a reusable skill from the trace — when to apply it and the concrete steps that worked.

skill → skill_bank
3

Inject & improve

Next task, relevant skills are retrieved semantically and injected. Same model — the agent stops repeating the encoded failure, measured against a no-skill control.

retrieve → inject → ↑
03what's inside

Built like a real system

Every piece of the loop is a first-class component, with governance, retrieval, and evaluation baked in.

Self-evolving skills

Skills aren't static. They version, merge, and improve from each new execution that touches them.

Semantic retrieval

05questions

Straight answers

Do I need to fine-tune anything?

No. Skills are retrieved and injected into the prompt at runtime — the model's weights never change. Periodic fine-tuning (Layer 2) is optional; the core loop is fine-tune-free.

Does it work with my model?

Any LLM — Gemini, Llama, or any OpenAI-compatible endpoint. The skill layer is model-agnostic, so you can swap the underlying model without rebuilding your skills.

How do you know the skills actually help?

Every run is scored by an LLM judge against a no-skills control arm. The reported skill_delta is the measured effect of skills versus that control — not a before/after number that conflates "the model can already code" with "the skills helped."

Can a bad skill reach production?

No. Skills move through draft → pending → approved → active. You review and approve or reject before anything goes live.

Where does it run?

Headless. Drive the loop from the yunaki CLI or API, and wire it into CI or a cron. Self-host it — it's Apache 2.0.
